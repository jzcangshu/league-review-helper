# 入团申请材料审核助手

这是一个面向学校批量审核工作的本地 WebUI（网页界面）工具。用户只需要选择 PDF 文件夹和 Excel 名单，就可以在浏览器中完成资料导入、姓名核对、PDF 预览、审核结果记录、OCR（光学字符识别）辅助检查和 Excel 回填。

所有学生材料默认只在本机处理，不需要上传到第三方服务。

## 适用场景

- 一所或多所学校需要连续审核大量入团志愿书 PDF。
- PDF 文件名包含班级、年级、申请书类型等附加文字。
- Excel 名单与 PDF 姓名可能存在缺漏、错别字或二次送审结果。
- 审核人员不熟悉项目目录，也不希望手工创建 TXT 文件。
- 希望材料始终留在本地，同时使用 OCR 辅助定位必填内容。

## 快速开始

### 1. 准备运行环境

当前版本优先支持 Windows 10/11。

普通用户不需要提前安装 Node.js 或 Python。首次双击启动时，程序会：

- 优先使用电脑中已有的 Node.js 18+ 和 Python 3.10+。
- 缺少 Node.js 时，自动从官方网站下载便携版到软件目录。
- 缺少 Python 时，自动从官方网站安装到软件自己的本地运行目录。
- 自动安装网页依赖；首次启用 OCR 时继续安装 OCR 依赖并下载模型。

整个过程不需要管理员权限，也不会修改系统 PATH（环境变量）。首次准备需要：

- 可访问 Node.js、Python 软件包和 OCR 模型下载地址的网络。
- 建议预留至少 1 GB 可用空间。

自动下载的基础运行环境保存在 `review-web/.runtime`，OCR 环境和模型分别保存在 `review-web/.ocr-python` 与 `review-web/.ocr-models`。

### 2. 下载并启动

从 GitHub Release（GitHub 发行版）下载源码压缩包并解压，然后双击：

~~~text
双击我启动审核软件.cmd
~~~

首次启动会自动准备缺少的运行环境和网页依赖，请保持窗口开启。启动完成后会打开本地地址：

~~~text
http://127.0.0.1:4173
~~~

如果 4173 端口已被占用，程序会自动选择后续可用端口。

也可以在 Windows Terminal（Windows 终端）中启动：

~~~powershell
cd review-web
npm install
npm start
~~~

## 在 WebUI 中导入学校

导入过程始终显示在左侧顶部，并按顺序引导完成。

### 第一步：选择 PDF 文件夹

选择实际存放 PDF 的文件夹。该文件夹的原始名称会直接作为本次导入的学校名称，程序不会向上推断或改写学校名。

PDF 文件名不需要只有姓名。程序会自动去除常见附加内容，例如：

- 班级前缀：801、803班、807班。
- 材料类型：入团申请书、入团志愿书、申请书、志愿书。
- 转换标记：转PDF、pdf。
- 空格、连接符和导出编号。

例如：

~~~text
803班 王宣淇 入团志愿书.pdf  ->  王宣淇
803彭馨凝pdf.pdf             ->  彭馨凝
806沈奕帆 .pdf               ->  沈奕帆
~~~

### 第二步：选择 Excel 名单

选择包含名单的 Excel 文件。程序会检查各工作表和前 80 行，综合表头含义、名单数据形态和相邻字段判断名单区域；支持“姓名”“学生姓名”“团员姓名”等常见写法，也支持最多三行的组合表头。

历史审核意见列按以下顺序识别：

1. 精确表头“问题备注”。
2. 精确表头“问题”。
3. 唯一一个包含“问题”的表头。
4. 精确表头“审核意见”“审核结果”或“审核备注”。
5. 精确表头“备注”。

第一张预览页顶部提供工作表、表头行、姓名列和审核意见列设置。用户可以原地调整后重新预览，不需要退出导入流程。存在多个候选、表头不明确或姓名列出现可疑内容时，程序会阻止继续操作，直到用户修正并确认。

### 第三步：核对导入内容

导入弹窗分为三个连续页面：

1. Excel 导入审核意见预览：以“姓名、问题”两列展示识别结果，并支持修正读取区域。被标红的异常姓名可以直接修改；确认导入时会先备份，再同步修正原 Excel。
2. 错漏人员自动识别：处理只在名单出现、只在 PDF 出现和疑似错别字。
3. 生成名单核对报告：生成可直接复制的纯文本核对结果。

疑似错别字必须人工选择真实姓名。界面支持打开对应 PDF，供用户查看首页手写姓名。

同一个 PDF 对应多个相似名单姓名时，界面会先要求选择疑似对应人员，再判断以名单姓名或 PDF 姓名为准；不会把所有组合平铺成多个独立问题。如果确认并非同一人的登记笔误，可以选择“并非同一人，保留两人”，程序会保留原名单人员，并把 PDF 姓名作为新人员补充到 Excel。

只有 PDF、未出现在名单中的人员默认选择新增到 Excel，也可以逐人改为不新增。程序会避免重复补充姓名。

### 第四步：确认导入

确认后程序会：

- 在 <code>审核结果/学校名</code> 下创建缺失的审核结果 TXT。
- 导入 Excel 中已有的历史审核意见。
- 保留已有的非空 TXT，不静默覆盖本地审核结果。
- 对确认使用 Excel 结果的冲突项先创建历史备份。
- 注册所选 PDF 文件夹，使学校立即出现在审核列表中。

已导入学校可以在“学校管理”中移除或恢复。移除只会隐藏学校，不会删除 PDF、Excel 或审核结果。

## 批量审核

### 页面布局

- 左侧：学校、学生、审核结果、审核注意事项和常用工具。
- 中间：PDF 缩略图与当前页面。
- 右侧：OCR 自动核验结果。

PDF 默认按完整显示一页的最大比例适配预览区。

### 审核结果

审核结果输入框会自动保存，不需要点击保存按钮。

- 有问题：输入文字或使用快捷短语。
- 无问题：保持输入框为空，并点击“确认无问题”。
- “确认无问题”会保存状态并自动切换到下一份。
- “标记为未审核”可以撤销明确审核状态。

程序会区分：

- 已审核且有问题：TXT 中有内容。
- 已审核且无问题：TXT 为空，并记录在 <code>.review-status.json</code>。
- 未审核：即使已经预创建空 TXT，也不会被视为审核完成。

### 快捷操作

- PgUp（向上翻页键）或 ↑：PDF 上一页。
- PgDn（向下翻页键）或 ↓：PDF 下一页。
- 小键盘 1-6：插入六条快捷短语。
- PDF 停留至少 10 秒且切换过页码后，再切换学生时会自动标记为已审核。
- 快捷短语与 Excel 回填共用底部工具区域，可通过页签切换。

翻页键由页面全局接管，即使鼠标焦点位于审核结果输入框之外也可以使用。

### 审核注意事项

左侧注意事项可以直接在网页中编辑。保存前会保留历史版本，当前内容存放在 <code>注意事项.txt</code>。

编辑时每行代表一项，页面会自动添加连续编号，不需要在文本中手写序号。<code>【重点】</code> 或 <code>**重点**</code> 会显示为加粗文字，<code>*OCR 提示*</code> 会显示为红色正体提示文字。

## OCR 辅助核验

OCR 功能默认开启，可以使用工具栏中的“OCR 标注”随时关闭。

首次使用高级 OCR 时，程序会自动：

1. 创建独立 Python 虚拟环境。
2. 安装 PaddleOCR（飞桨文字识别）、ONNX Runtime（ONNX 推理运行环境）和 PyMuPDF（PDF 处理库）。
3. 下载 PP-OCRv6_small_det（文字检测模型）和 PP-OCRv6_tiny_rec（文字识别模型）。

首次准备时间取决于网络速度，之后会复用本地环境、模型和识别缓存。审核当前 PDF 时，程序会预识别后两份资料。

### 自动核验项目

右侧竖栏展示五项结果：

- 年龄门槛：识别出生年月，计算首次团课不可早于的月份。
- 团课学时：优先统计证明人记录，辅助统计“1、1h、1学时”，要求不少于 8 学时。
- 团纪处分条例：团课内容中命中“团纪、纪律、处分、条例”任一词即视为存在相关课程。
- 信仰声明：分别核对志愿页、两位介绍人和支部页面中的声明数量。
- 积极分子提醒：检测到相关表述时，提醒人工比较积极分子或递交申请时间与首次团课。

年龄门槛和积极分子项目提供两个页面直达按钮，便于来回人工核对。

### 高光颜色

- 蓝色：辅助定位信仰声明和连续出现的系列思想。
- 黄色：OCR 不完整或需要人工确认。
- 红色：明确缺失或不合规的位置。

高光采用低透明度填充，不使用描边，避免遮挡手写文字。

OCR 只用于辅助定位和提醒，最终结论仍应由审核人员确认。

## Excel 回填

在左侧底部切换到“5. 回填 Excel”，即可将当前学校的审核结果写回名单。

回填时程序会：

- 保留原 Excel 不变，在同一文件夹生成带日期时间的审核回填副本，例如 <code>名单_7月14日_150607_审核回填.xlsx</code>。
- 覆写已审核人员的结果。
- 已审无问题写入空白单元格。
- 未审核的空 TXT 以黑色写入“未审核”。
- 只有没有对应 PDF 的人员以红色写入“无资料”。
- 补充确认需要新增、且具有精确 PDF 的人员。
- 按姓名去重，避免重复新增。
- 高置信姓名差异仅在候选唯一时自动对应。
- 回填按钮会直接显示绿色成功或红色失败状态；成功后同时显示已审、未审、无 PDF 和新增人数，并提供“打开 Excel 文件”和“打开所在文件夹”。

回填会复用导入时确认过的工作表、表头和列设置。工作簿结构发生变化或仍存在可疑姓名时会停止写入，不会猜测目标区域。

## 姓名匹配原则

导入分析、审核列表和 Excel 回填使用同一套姓名规则：

1. 优先精确匹配清洗后的姓名。
2. 允许文件名包含姓名或姓名包含清洗结果。
3. 同长度、同姓、仅一字不同且候选唯一时，视为高置信疑似错别字。
4. 多候选或低置信情况不自动猜测，交给用户处理。
5. 同一 PDF 在不同的用户导入源中独立匹配，不会被先加载的学校独占。

所有姓名纠正都在导入核对页面中显示，用户可以选择以 Excel 姓名或 PDF 姓名为准。

主要本地数据：

~~~text

    姓名_审核结果.txt
    .review-status.json

  .ocr-cache/

  .ocr-models/
~~~

- <code>审核结果历史</code>：保存被替换的审核结果。
- <code>Excel历史</code>：保存导入核对过程中改名或修正名单前的工作簿；整校回填使用同目录副本，不覆盖原文件。
- <code>注意事项历史</code>：保存网页内编辑前的旧版本。
- <code>sources.local.json</code>：保存用户导入的学校路径。
- OCR 目录都是本地运行缓存，可以重新生成。

程序不会主动删除用户的 PDF、Excel 或审核结果。

## 可选的 Codex Skill

WebUI 已经能够独立完成导入、核对、审核和回填。<code>league-review-prep</code> 技能适合需要命令行批处理、无人值守准备或由 Codex 代为操作的场景。

审核前准备：

~~~powershell
python codex-skills/league-review-prep/scripts/prepare_school_review.py --school-dir "学校文件夹" --update-web-sources
~~~

审核后回写：

~~~powershell
python codex-skills/league-review-prep/scripts/prepare_school_review.py --school-dir "学校文件夹" --write-excel --result-column "问题备注"
~~~

技能脚本遵循与 WebUI 相同的姓名匹配和审核状态规则，不会创建不存在的 Excel 结果列。

特殊工作簿可以明确指定布局：

~~~powershell
python codex-skills/league-review-prep/scripts/prepare_school_review.py --excel "学校/名单.xlsx" --sheet "名单" --header-row 3 --name-column B --pdf-dir "学校/资料目录"
~~~

## 常见问题

### 双击启动没有反应

请使用根目录的 `双击我启动审核软件.cmd`。启动失败时窗口会停留并显示错误，不会自动关闭。

服务自身的错误还会写入：

~~~text
review-web/start-review-error.log
review-web/start-review-output.log
~~~

也可以在项目目录打开 Windows Terminal（Windows 终端），运行：

~~~powershell
powershell -NoProfile -ExecutionPolicy Bypass -File .\start-review.ps1
~~~

首次自动准备环境需要联网。下载失败时可以直接再次双击，程序会继续复用已经下载或安装成功的内容。

也可以进入 <code>review-web</code> 后运行 <code>npm start</code> 查看错误。

### 选择 PDF 文件夹后显示缺 PDF

确认选择的是直接或递归包含 PDF 的文件夹。文件名可以包含班级和材料类型，不需要手工改成纯姓名。

如果同一目录被多个学校源引用，每个源会独立匹配，不会互相抢占。

### OCR 首次运行很慢

首次运行需要安装 Python 依赖并下载模型。等待右侧进度完成即可。之后相同 PDF 会读取缓存。

### OCR 识别结果不可靠

扫描模糊、倾斜、合页或手写过于潦草都会影响识别。黄色结果表示需要人工核对；OCR 通过也不替代人工查看原页。

### Excel 回填列不正确

在导入预览第一页检查工作表、表头行、姓名列和审核意见列。程序依次识别“问题备注”“问题”、唯一含“问题”的列、审核意见别名和精确“备注”。结构不确定时应在预览中明确选择，不要通过重命名文件绕过确认。

### 端口被占用

程序会自动尝试后续端口。实际端口记录在 <code>review-web/review-web-port.txt</code> 和系统临时目录的端口文件中。

## 开发与测试

安装依赖：

~~~powershell
cd review-web
npm install
~~~

运行测试：

~~~powershell
npm test
~~~

启动开发服务：

~~~powershell
npm start
~~~

主要目录：

~~~text
review-web/server.js                 本地 HTTP 服务
review-web/public/                   WebUI
review-web/lib/import-service.js     导入、名单核对和历史结果处理
review-web/lib/export-service.js     Excel 回填
review-web/lib/ocr-service.js        OCR 环境、模型与缓存
codex-skills/league-review-prep/     可选 Codex 技能
审核结果/                            审核结果根目录
注意事项.txt                         默认审核注意事项
~~~

服务端使用 Node.js 内置 HTTP 模块。前端不依赖大型框架，PDF 预览使用 PDF.js（PDF 渲染库）。

## 隐私与开源边界

真实学生 PDF、Excel、审核结果、身份证号和联系方式不应提交到公开仓库。

项目的 <code>.gitignore</code> 使用顶层允许列表，并默认排除：

- PDF 和 Excel 文件。
- 本地学校源配置。
- OCR 模型、虚拟环境与缓存。
- Node.js 和 Python 依赖缓存。

发布前应检查 <code>git status --short</code>，确认仓库中只包含示例资料和程序文件。
